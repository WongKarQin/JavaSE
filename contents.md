# JavaSE 与 JavaEE 知识与技术大纲

说明：本大纲以“知识点 - 该技术解决了什么问题，可能存在什么不足，使用时需要注意什么，常用于什么场景”为主线，适合作为学习路线、复习目录和项目查漏补缺清单。

## 一、JavaSE 基础

- Java 语言特性：解决跨平台、面向对象、自动内存管理的问题；不足是语法和生态较大；注意理解 JVM、JRE、JDK 的职责边界；常用于企业应用、服务端开发、大数据与 Android 基础学习。
- 开发环境与工具链：解决代码编写、编译、运行、调试的问题；不足是初学阶段配置容易混乱；注意 `PATH`、编码、项目结构与构建工具版本；常用于本地开发、调试、打包发布。
- 数据类型：解决不同数据表达和存储的问题；不足是基本类型存在范围限制、浮点数有精度误差；注意自动类型提升、强转、包装类与空值问题；常用于变量定义、运算与接口数据承载。
- 运算符：解决算术、关系、逻辑、位运算的问题；不足是优先级易出错；注意短路逻辑、自增副作用、位运算使用边界；常用于条件判断、状态标记、权限位设计。
- 流程控制：解决顺序、分支、循环执行的问题；不足是嵌套过深会降低可读性；注意 `break`、`continue`、`switch` 穿透和异常分支；常用于业务判断、数据遍历、流程编排。
- 数组：解决同类型数据批量存储的问题；不足是长度固定、操作能力有限；注意下标越界、默认值和多维数组本质；常用于基础数据处理、算法题和底层容器实现理解。
- 方法：解决代码复用、职责拆分的问题；不足是切分过细会增加理解成本；注意参数传递本质、返回值设计、命名语义；常用于业务封装、工具方法与模块抽象。
- 重载：解决相同语义下参数形式不同的问题；不足是重载过多会影响可读性；注意匹配优先级、可变参数与自动装箱干扰；常用于工具类、构造器、API 友好设计。
- 面向对象：解决现实世界建模和复杂系统抽象的问题；不足是设计不当会导致层次混乱；注意对象职责、封装边界和依赖方向；常用于业务建模、框架设计和系统分层。
- 封装：解决内部实现暴露过多的问题；不足是封装过度会增加调用成本；注意访问修饰符、属性校验和不变量保护；常用于实体类、组件设计、模块边界控制。
- 继承：解决代码复用和层次扩展的问题；不足是耦合强、易形成脆弱基类；注意“is-a”关系、父类构造过程和方法覆写规则；常用于模板代码复用、框架扩展点设计。
- 多态：解决统一接口调用不同实现的问题；不足是运行期行为不直观、调试复杂；注意向上转型、动态绑定和接口优先于继承；常用于策略模式、SPI、面向接口编程。
- `this` 与 `super`：解决当前对象与父类成员访问的问题；不足是滥用会让代码绕；注意构造器调用顺序和同名成员区分；常用于构造器链、覆写方法和父类能力复用。
- 抽象类：解决共性代码复用与统一约束的问题；不足是单继承限制扩展性；注意抽象类聚合公共状态，子类完成差异实现；常用于模板模式、领域基类、框架骨架。
- 接口：解决能力约定和解耦协作的问题；不足是接口过多会增加理解成本；注意面向接口编程、默认方法边界和版本兼容；常用于服务定义、策略扩展、模块解耦。
- 枚举：解决固定取值集合表达不清的问题；不足是设计不当会承载过多业务逻辑；注意枚举可带属性和方法，但不要过度复杂；常用于状态机、类型分类、配置常量。
- 包机制与访问控制：解决命名冲突和代码组织的问题；不足是包划分不合理会导致结构混乱；注意 `public`、`protected`、默认、`private` 的可见性；常用于模块分层和 API 暴露控制。
- 常用类 `Object` / `String` / `StringBuilder`：解决对象共性、字符串处理性能与不可变性的问题；不足是字符串拼接不当会浪费内存；注意 `equals`、`hashCode`、不可变对象与线程安全；常用于日志、文本处理、键值设计。
- 日期时间 API：解决旧版时间类易错、线程不安全的问题；不足是时区和格式化仍有学习成本；注意优先使用 `java.time` 包，明确时区与格式；常用于订单时间、调度、日志分析。
- 数学与随机工具类：解决数值运算和随机数据生成的问题；不足是伪随机不适合安全场景；注意 `BigDecimal` 精度、舍入模式和安全随机数选择；常用于金额计算、模拟数据、统计分析。

## 二、集合与泛型

- 集合框架总览：解决对象批量存储、检索、去重、排序的问题；不足是不同容器复杂度差异大；注意根据访问特征选择结构；常用于业务数据缓存、批量处理、内存模型设计。
- `List`：解决有序可重复集合的问题；不足是中间插入删除可能性能差；注意 `ArrayList` 与 `LinkedList` 的适用场景；常用于结果集承载、顺序任务、页面列表。
- `Set`：解决元素去重的问题；不足是无序或排序成本存在差异；注意去重依赖 `equals` / `hashCode` 或比较器；常用于标签集合、ID 去重、权限集合。
- `Map`：解决键值映射查询的问题；不足是键设计不当会造成覆盖或查找异常；注意键不可变性、空键空值规则与遍历方式；常用于缓存、配置、索引与统计。
- `Collections` / `Arrays` 工具类：解决排序、查找、转换的问题；不足是原地修改可能带来副作用；注意不可变集合和视图集合区别；常用于快速处理集合数据。
- 泛型：解决类型安全和强制转换过多的问题；不足是类型擦除会限制部分运行期能力；注意通配符上下界、泛型方法和不能直接创建泛型数组；常用于公共组件、集合 API、数据封装。
- 迭代器与 `for-each`：解决统一遍历的问题；不足是并发修改易报错；注意 `Iterator` 删除元素的正确方式；常用于集合遍历、过滤和批处理。
- 比较器 `Comparable` / `Comparator`：解决对象排序规则定义的问题；不足是比较规则不一致会导致集合行为异常；注意自反性、传递性与空值处理；常用于排行榜、订单排序、搜索结果排序。

## 三、异常、调试与测试

- 异常体系：解决错误与正常流程分离的问题；不足是滥用异常会影响性能和可读性；注意受检异常与运行时异常的取舍；常用于参数校验、资源访问、业务失败处理。
- 异常处理：解决错误定位、恢复与上抛的问题；不足是空 `catch` 会隐藏问题；注意异常链、统一封装和日志记录；常用于服务层、控制层与任务调度。
- 自定义异常：解决业务语义表达不清的问题；不足是层级过多会增加维护成本；注意错误码、上下文信息和统一基类；常用于订单、支付、权限等领域异常。
- 日志：解决问题排查和运行态观察的问题；不足是日志过多会影响性能和检索效率；注意日志级别、脱敏、链路标识和结构化输出；常用于线上排障、审计、监控分析。
- 单元测试：解决代码回归验证的问题；不足是测试质量依赖用例设计；注意边界值、异常分支和可重复执行；常用于工具类、业务逻辑、重构保护。
- 断点调试：解决复杂流程定位的问题；不足是对并发和生产问题帮助有限；注意条件断点、表达式求值与变量跟踪；常用于本地排错和流程理解。

## 四、I/O、网络与并发

- I/O 流：解决文件、网络、控制台数据读写的问题；不足是流层次多、易混淆；注意字节流与字符流、缓冲流与关闭顺序；常用于文件处理、日志导出、对象传输。
- NIO / NIO.2：解决高性能 I/O 和异步通道的问题；不足是学习门槛较高；注意 `Buffer` 状态切换、`Channel`、选择器与文件 API；常用于网关、通信框架、大文件处理。
- 序列化：解决对象持久化与传输的问题；不足是版本兼容、安全和性能问题明显；注意 `serialVersionUID` 与敏感字段；常用于缓存、消息传递、历史系统对接。
- 网络编程：解决进程间通信的问题；不足是协议与网络异常处理复杂；注意 TCP/UDP 差异、粘包拆包和超时重试；常用于聊天、RPC、物联网接入。
- 线程基础：解决程序并发执行的问题；不足是共享资源竞争容易出错；注意线程生命周期、中断机制和不要直接滥建线程；常用于异步任务、并行计算、后台处理。
- 线程同步：解决并发安全和数据一致性的问题；不足是锁竞争会影响吞吐量；注意 `synchronized`、`volatile`、`Lock`、可见性与原子性；常用于库存扣减、状态更新、缓存刷新。
- 线程池：解决线程重复创建成本高和资源失控的问题；不足是参数配置不合理会引发堆积或拒绝；注意核心线程数、队列长度、拒绝策略；常用于 Web 请求处理、异步通知、批处理任务。
- 并发工具类：解决计数、阻塞协作、并发容器等问题；不足是误用会引起死锁或性能问题；注意 `CountDownLatch`、`CyclicBarrier`、`Semaphore`、`ConcurrentHashMap` 的语义；常用于限流、并行任务编排、缓存。

## 五、JVM 与高级特性

- 反射：解决运行期动态获取和操作类信息的问题；不足是性能较低、可读性差且破坏封装；注意缓存反射结果和访问权限控制；常用于框架开发、注解解析、IOC 容器。
- 注解：解决元数据声明和配置外置的问题；不足是滥用会让行为变隐式；注意保留策略、作用目标和处理时机；常用于 Spring、ORM、校验、测试框架。
- Lambda 与函数式编程：解决匿名内部类冗长的问题；不足是调试堆栈不如显式代码直观；注意函数式接口、变量捕获和可读性；常用于集合处理、回调、策略传递。
- Stream API：解决集合声明式处理的问题；不足是复杂链式调用不易调试；注意惰性执行、副作用控制和并行流使用边界；常用于过滤、映射、分组、聚合统计。
- JVM 内存模型：解决对象创建、回收、线程可见性理解的问题；不足是概念抽象；注意堆、栈、方法区、直接内存及逃逸分析；常用于性能调优、线上问题排查。
- 垃圾回收 GC：解决内存自动回收的问题；不足是停顿不可完全避免；注意垃圾收集器选择、对象晋升和 Full GC 风险；常用于高并发服务优化和内存故障分析。
- 类加载机制：解决类按需加载和扩展隔离的问题；不足是双亲委派和类冲突排查较难；注意自定义类加载器、热加载边界和 SPI；常用于容器、插件化、框架设计。
- 性能调优：解决吞吐量、延迟和资源利用率不佳的问题；不足是调优需要数据支撑；注意先监控后优化、定位瓶颈再改代码或参数；常用于线上服务优化、批任务提速。

## 六、数据库与持久化基础

- SQL 基础：解决结构化数据增删改查的问题；不足是复杂查询维护成本高；注意索引、事务、连接方式与执行计划；常用于业务数据存储、统计报表、数据治理。
- 关系型数据库设计：解决数据结构规范化和一致性的问题；不足是过度范式化会影响查询性能；注意主键、外键、冗余平衡和字段语义；常用于订单、用户、支付等核心系统。
- JDBC：解决 Java 直接访问数据库的问题；不足是样板代码多、手工映射繁琐；注意连接释放、预编译、防 SQL 注入和事务控制；常用于底层数据库访问和框架原理学习。
- 事务基础：解决多步操作一致性的问题；不足是长事务会影响并发；注意 ACID、隔离级别、脏读与幻读；常用于转账、下单、库存与积分变更。
- 连接池：解决数据库连接创建成本高的问题；不足是配置不当会耗尽资源；注意池大小、超时、泄漏检测与监控；常用于所有数据库驱动的企业应用。

## 七、Java Web / JavaEE 基础

- HTTP 协议：解决浏览器与服务器通信的问题；不足是无状态；注意请求方法、状态码、头信息、幂等性和缓存；常用于 Web 开发、接口设计、抓包排错。
- 前端基础 HTML / CSS / JavaScript：解决页面结构、样式、交互的问题；不足是与 Java 后端有技术栈边界；注意表单提交、AJAX、JSON 与跨域；常用于 JSP 页面、后台管理系统和前后端联调。
- XML：解决结构化配置和数据交换的问题；不足是冗长；注意命名空间、DTD/XSD 和解析方式；常用于旧系统配置、MyBatis 映射、Web 配置。
- JSON：解决轻量级数据交换的问题；不足是类型约束弱；注意序列化格式、空值处理和时间字段规范；常用于前后端通信、REST 接口、消息体传输。
- Servlet：解决 Java Web 请求处理的问题；不足是原生开发样板代码多；注意生命周期、线程安全、请求转发与重定向；常用于学习 Web 原理、老项目维护。
- Filter：解决横切逻辑统一处理的问题；不足是链路过多会影响排查；注意执行顺序、放行与编码处理；常用于登录校验、权限控制、日志、跨域。
- Listener：解决容器事件监听的问题；不足是使用面较窄；注意会话、上下文、请求事件的生命周期；常用于初始化资源、在线人数统计、会话管理。
- JSP：解决动态页面渲染的问题；不足是表现层与逻辑易耦合；注意尽量少写脚本片段，配合 JSTL/EL 使用；常用于传统后台页面和教学项目。
- JSTL / EL：解决 JSP 页面逻辑表达混乱的问题；不足是复杂逻辑能力有限；注意只做展示层表达，不承载业务；常用于页面循环、判断、格式化输出。
- Cookie：解决客户端状态保存的问题；不足是容量小且可被篡改；注意过期时间、作用域、安全属性；常用于记住登录、偏好设置、会话标识。
- Session：解决服务端会话状态管理的问题；不足是分布式场景共享成本高；注意超时、序列化和 Session 固定攻击防护；常用于登录态、购物车、临时状态保存。

## 八、Spring 生态

- Spring IOC：解决对象创建和依赖管理耦合过高的问题；不足是过度配置会增加理解成本；注意 Bean 生命周期、作用域和依赖注入方式；常用于整合业务组件、基础设施组件。
- Spring AOP：解决日志、事务、权限等横切逻辑散落的问题；不足是调用链变隐式；注意代理机制、切点表达式和自调用失效；常用于日志审计、事务、权限、监控。
- Spring 事务：解决声明式事务管理的问题；不足是传播行为和异常回滚规则容易误判；注意回滚条件、传播属性和方法可见性；常用于数据库一致性控制。
- SpringMVC：解决 Web 层请求分发和参数绑定的问题；不足是理解执行链需要时间；注意控制器职责、参数校验、拦截器和异常处理；常用于接口开发、页面跳转、REST 服务。
- RESTful API：解决接口风格统一的问题；不足是设计不规范会名不副实；注意资源命名、状态码、幂等性、分页与版本管理；常用于前后端分离和开放接口。
- 参数校验：解决接口输入不可信的问题；不足是规则分散时维护困难；注意 JSR-303、分组校验和统一错误响应；常用于表单提交、API 入参校验。
- 全局异常处理：解决控制层异常返回不统一的问题；不足是过度包装会掩盖根因；注意日志与用户提示分离；常用于统一接口响应、错误码体系建设。
- 拦截器：解决请求前后统一处理的问题；不足是滥用会使链路复杂；注意与 Filter 的职责区分；常用于鉴权、日志、性能统计、上下文注入。

## 九、Spring Boot 与配置化开发

- Spring Boot：解决 Spring 项目搭建复杂、整合繁琐的问题；不足是自动配置过多时排障不直观；注意约定优于配置、启动类扫描范围和依赖版本管理；常用于快速构建微服务与单体应用。
- 自动配置：解决第三方组件集成成本高的问题；不足是默认行为不理解时易踩坑；注意条件装配、配置优先级和 `starter` 机制；常用于中间件快速接入。
- 配置管理：解决多环境参数管理的问题；不足是配置分散会提升维护成本；注意 `application.yml`、环境隔离、敏感信息外置；常用于开发、测试、生产切换。
- 日志与监控端点：解决应用可观测性不足的问题；不足是暴露过多信息有安全风险；注意 Actuator 暴露范围和权限控制；常用于健康检查、指标采集、故障定位。
- 定时任务：解决周期性任务执行的问题；不足是分布式场景可能重复执行；注意时区、幂等和任务互斥；常用于报表生成、缓存刷新、数据同步。
- 热部署与开发效率工具：解决开发重启成本高的问题；不足是生产不可依赖；注意仅用于开发环境；常用于本地联调、快速迭代。

## 十、持久层框架

- MyBatis：解决 JDBC 样板代码过多和 SQL 映射的问题；不足是复杂 SQL 与 XML 维护成本较高；注意参数绑定、结果映射、动态 SQL 与缓存边界；常用于对 SQL 可控要求高的项目。
- MyBatis 全局配置：解决别名、映射、日志、缓存等统一配置的问题；不足是配置分散时排查困难；注意驼峰映射、懒加载和日志输出；常用于项目基础设施搭建。
- 动态 SQL：解决查询条件变化多的问题；不足是复杂条件可读性差；注意判空、SQL 片段复用和注入风险；常用于后台检索、组合查询、报表过滤。
- MyBatis 缓存：解决重复查询性能问题；不足是一二级缓存一致性控制复杂；注意更新场景缓存失效和命中预期；常用于读多写少场景。
- MyBatis Plus：解决通用 CRUD 编写重复的问题；不足是复杂业务不能完全依赖自动生成；注意条件构造器、分页插件和代码生成器边界；常用于后台管理系统、标准化业务模块。

## 十一、安全、权限与中间件整合

- Spring Security：解决认证授权和安全控制的问题；不足是学习曲线较陡；注意过滤器链、用户体系、密码加密和资源权限规则；常用于登录鉴权、接口保护、单点登录基础。
- Shiro：解决认证授权与会话控制的问题；不足是近年在新项目中活跃度不如 Spring Security；注意与 Spring 整合方式和过滤规则；常用于老系统维护或轻量权限方案。
- Redis：解决高性能缓存、分布式锁和临时数据存储的问题；不足是缓存一致性与内存淘汰策略需要设计；注意键设计、过期策略和热点问题；常用于缓存、验证码、排行榜、分布式会话。
- 消息队列：解决系统解耦、削峰填谷和异步处理的问题；不足是引入最终一致性和消息可靠性问题；注意重复消费、顺序性和死信处理；常用于订单异步通知、日志收集、事件驱动。

## 十二、工程化与软件工程

- Maven：解决依赖管理和项目构建的问题；不足是配置复杂时理解成本高；注意生命周期、坐标、依赖传递和仓库镜像；常用于 Java 项目标准构建。
- Gradle：解决更灵活的构建编排问题；不足是团队需要统一 DSL 风格；注意插件版本、任务依赖和缓存机制；常用于 Android、复杂多模块项目和现代构建场景。
- 多模块项目：解决大型系统拆分和复用的问题；不足是模块边界不清会导致耦合；注意聚合、继承、公共模块与循环依赖；常用于中后台系统、微服务前置拆分。
- Git：解决版本管理和协作开发的问题；不足是分支策略不规范会混乱；注意提交信息、冲突处理和分支模型；常用于团队协作、代码审查、回滚追踪。
- 设计模式：解决常见设计问题的复用方案问题；不足是生搬硬套会过度设计；注意先识别变化点再选模式；常用于框架设计、代码重构、复杂业务抽象。
- 软件设计原则：解决高耦合、低扩展的问题；不足是原则之间需要平衡；注意单一职责、开闭原则、里氏替换、依赖倒置、接口隔离、迪米特法则；常用于架构设计和重构。
- UML：解决系统结构和流程表达不清的问题；不足是维护成本存在；注意只画有价值的图；常用于需求澄清、设计评审、文档沉淀。
- 软件测试：解决质量验证和回归保障的问题；不足是覆盖率不代表质量；注意单测、集成测试、系统测试、压力测试和测试数据隔离；常用于提测前校验与持续交付。
- 质量管理与代码规范：解决团队代码风格不一致和缺陷率高的问题；不足是规则过多会影响开发体验；注意静态检查、Review、命名和复杂度控制；常用于团队协作和长期维护。
- 项目管理与估算：解决需求、进度、风险不可控的问题；不足是估算永远存在不确定性；注意里程碑、风险前置和需求变更控制；常用于课程设计、团队项目与企业研发。

## 十三、学习与落地建议

- 学习顺序建议：先打牢 JavaSE，再学数据库与 Web 基础，再进入 Spring 生态和工程化；不足是跳学框架容易“会配不会用”；注意每学一块都用小项目验证；常用于建立完整知识地图。
- 项目实践建议：从控制台程序到 Servlet/JSP，再到 SpringBoot + MyBatis + Redis 的后台项目逐步升级；不足是只看不练容易遗忘；注意需求拆分、日志、异常、测试和部署；常用于求职准备和课程实战。
- 复习方法建议：按“概念、解决问题、核心 API、常见坑、面试题、项目场景”六个维度复盘；不足是只背八股容易脱离实战；注意把知识点映射到真实代码；常用于期末复习、面试冲刺和知识查漏补缺。
